3.1	Create work_threads (the most critical new table)	Supabase SQL Editor

Creates work_threads and backfills one thread per unique job_request that had messages. The 4 active conversations (2 unique request IDs with messages) will each get a thread.

SQL — Supabase SQL Editor
CREATE TABLE IF NOT EXISTS work_threads (
  id             UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  job_request_id UUID REFERENCES job_requests(id) ON DELETE SET NULL,
  client_id      UUID NOT NULL REFERENCES profiles(id),
  provider_id    UUID NOT NULL REFERENCES profiles(id),
  status         TEXT NOT NULL DEFAULT 'new'
                   CHECK (status IN ('new','quoted','negotiating','active',
                                     'completed','reviewed','disputed')),
  created_at     TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at     TIMESTAMPTZ NOT NULL DEFAULT now()
);

ALTER TABLE work_threads ENABLE ROW LEVEL SECURITY;

CREATE POLICY "Participants view their threads"
  ON work_threads FOR SELECT
  USING (client_id = auth.uid() OR provider_id = auth.uid());

CREATE POLICY "Authenticated users create threads"
  ON work_threads FOR INSERT
  WITH CHECK (auth.uid() IS NOT NULL);

CREATE POLICY "Participants update their threads"
  ON work_threads FOR UPDATE
  USING (client_id = auth.uid() OR provider_id = auth.uid());

ALTER PUBLICATION supabase_realtime ADD TABLE work_threads;

-- Backfill: create a thread for each job_request that has messages.
-- Uses the accepted quote's provider as provider_id.
-- Falls back to the first quote if no accepted quote exists.
INSERT INTO work_threads (job_request_id, client_id, provider_id, status)
SELECT DISTINCT ON (jr.id)
  jr.id         AS job_request_id,
  jr.client_id  AS client_id,
  q.provider_id AS provider_id,
  CASE jr.status
    WHEN 'completed' THEN 'completed'
    WHEN 'pending'   THEN 'negotiating'
    ELSE 'quoted'
  END           AS status
FROM job_requests jr
JOIN quotes q ON q.request_id = jr.id
WHERE jr.id IN (SELECT DISTINCT request_id FROM direct_messages)
ORDER BY jr.id,
  CASE q.status WHEN 'accepted' THEN 0 ELSE 1 END,
  q.created_at ASC;

4.1	Enable PostGIS and create the match_providers() function	Supabase SQL Editor

PostGIS enables geo queries. The match_providers() SQL function will be called by the Lovable job request flow to find nearby Providers.

SQL — Supabase SQL Editor
-- Enable PostGIS (one-time per project)
CREATE EXTENSION IF NOT EXISTS postgis;

-- Add geometry column to provider_profiles
ALTER TABLE provider_profiles
  ADD COLUMN IF NOT EXISTS geom GEOMETRY(Point, 4326);

-- Spatial index for fast radius queries
CREATE INDEX IF NOT EXISTS provider_profiles_geom_idx
  ON provider_profiles USING GIST (geom);

-- Provider matching function
CREATE OR REPLACE FUNCTION match_providers(
  job_lat     DOUBLE PRECISION,
  job_lng     DOUBLE PRECISION,
  job_cat     TEXT,
  max_results INTEGER DEFAULT 8
)
RETURNS TABLE (
  provider_id  UUID,
  distance_km  DOUBLE PRECISION,
  score        DOUBLE PRECISION
) AS $$
BEGIN
  RETURN QUERY
  SELECT
    pp.user_id AS provider_id,
    ST_Distance(
      pp.geom::geography,
      ST_SetSRID(ST_MakePoint(job_lng, job_lat), 4326)::geography
    ) / 1000.0 AS distance_km,
    (
      COALESCE(pp.avg_rating, 0) * 0.4 +
      (1.0 / NULLIF(pp.response_time_minutes, 0)) * 0.3 * 100 +
      (1.0 / NULLIF(
        ST_Distance(
          pp.geom::geography,
          ST_SetSRID(ST_MakePoint(job_lng, job_lat), 4326)::geography
        ) / 1000.0, 0)) * 0.3
    ) AS score
  FROM provider_profiles pp
  WHERE
    job_cat = ANY(pp.categories)
    AND pp.geom IS NOT NULL
    AND ST_DWithin(
      pp.geom::geography,
      ST_SetSRID(ST_MakePoint(job_lng, job_lat), 4326)::geography,
      pp.service_radius_km * 1000
    )
  ORDER BY score DESC
  LIMIT max_results;
END;
$$ LANGUAGE plpgsql;

6.3	Create and deploy the STK Push Edge Function	VS Code


TERMINAL — scaffold the function
# Create the Edge Function scaffold
supabase functions new mpesa-stk-push

LOVABLE PROMPT — generate the Edge Function code
Build the Supabase Edge Function for M-Pesa STK Push payments.

Create file: supabase/functions/mpesa-stk-push/index.ts

The function should:
1. Accept POST with JSON body: { work_thread_id, amount_kes, phone_number }
2. Format phone: strip leading 0, prepend 254 (e.g. 0712345678 ? 254712345678)
3. Generate OAuth token:
   POST https://sandbox.safaricom.co.ke/oauth/v1/generate?grant_type=client_credentials
   Auth: Basic base64(CONSUMER_KEY:CONSUMER_SECRET)
4. Build STK Push body:
   - Timestamp: new Date() formatted as YYYYMMDDHHmmss
   - Password: base64(MPESA_SHORTCODE + MPESA_PASSKEY + Timestamp)
   - BusinessShortCode: Deno.env.get('MPESA_SHORTCODE')
   - TransactionType: "CustomerPayBillOnline"
   - Amount: Math.round(amount_kes) — must be integer
   - PartyA: formatted phone (2547XXXXXXXX)
   - PartyB: Deno.env.get('MPESA_SHORTCODE')
   - PhoneNumber: formatted phone
   - CallBackURL: Deno.env.get('MPESA_CALLBACK_URL')
   - AccountReference: "WorkPin"
   - TransactionDesc: "WorkPin Service Payment"
5. POST to: https://sandbox.safaricom.co.ke/mpesa/stkpush/v1/processrequest
   (use production URL when MPESA_ENV = 'production')
6. Save the CheckoutRequestID to the payments table
7. Return { success: true, checkoutRequestId: string }

Use Deno.env.get() for all credentials. Never hardcode.
Use the Supabase client from supabase-js to update the database.

6.4	Create and deploy the M-Pesa Callback Edge Function	VS Code

This is the public endpoint Safaricom calls when a payment completes or fails. It updates the payment record and WorkThread status automatically.

LOVABLE PROMPT — generate callback function
Build the Supabase Edge Function for M-Pesa payment callbacks.

Create file: supabase/functions/mpesa-callback/index.ts

The function should:
1. Accept POST requests from Safaricom Daraja (no auth required — it's public)
2. Parse the Daraja callback body:
   { Body: { stkCallback: { MerchantRequestID, CheckoutRequestID,
     ResultCode, ResultDesc, CallbackMetadata } } }
3. Find the payment record by mpesa_checkout_request_id = CheckoutRequestID
4. If ResultCode === 0 (success):
   a. Extract MpesaReceiptNumber from CallbackMetadata.Item array
   b. Update payments: status='paid', mpesa_receipt_number=receipt
   c. Update work_threads: status='completed', updated_at=now()
      WHERE id = payment.work_thread_id
   d. Insert a SYSTEM message in messages:
      body = 'Malipo ya KES [amount] yamepokelewa! M-Pesa Receipt: [receipt]'
      type = 'system', sender_id = client_id of the thread
5. If ResultCode !== 0 (failure):
   a. Update payments: status='failed'
   b. Insert SYSTEM message: 'Malipo yameshindwa. Tafadhali jaribu tena.'
6. Return HTTP 200 (Daraja requires 200 or it will retry)

TERMINAL — deploy (--no-verify-jwt because Daraja calls this without auth)
supabase functions deploy mpesa-callback --no-verify-jwt

?	Function URL matches the MPESA_CALLBACK_URL secret set in Step 6.2
?	Test with a mock Daraja payload — confirm payment record updates and SYSTEM message is inserted
