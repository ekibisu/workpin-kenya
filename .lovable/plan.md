

# Re-Analysis: External Supabase State vs. Codebase

## What Happened

Previous changes were made against the **Lovable Cloud DB** (`ebfewhicwahypqabtpbs`), which has a simpler schema. The app is now correctly pointed at your **external Supabase** (`pvrvhjzuqejkgosmaljf`), which has a richer schema. Several "fixes" from earlier sessions actually broke things for the external DB.

## Key Schema Differences

The external DB's `services` table has columns the Lovable Cloud DB doesn't:
- `archetype` (text) — proper archetype key per service
- `description` (text) — human-readable service description
- `is_active` (boolean) — soft-delete flag
- `sort_order` (integer) — display ordering

The external DB's `messages` table already has: `id`, `work_thread_id`, `sender_id`, `content`, `created_at` — this matches what the code expects.

## What Needs Fixing

### 1. Update `types.ts` to match the external DB
The auto-generated `types.ts` reflects the Lovable Cloud schema, not the external one. The `services` table type is missing `archetype`, `description`, `is_active`, and `sort_order`. Since this file is normally auto-generated by Lovable Cloud (which we can't change), we need to manually update it to match the external DB schema.

### 2. Update `src/types/services.ts` 
The local `Service` interface was stripped down to match the Lovable Cloud schema. It needs `archetype`, `description`, `is_active`, and `sort_order` as proper (non-optional) fields since the external DB has them.

### 3. Fix `useServicesByArchetype` grouping logic
Line 60 in `useServices.ts` calls `normaliseArchetype(service.category, service.name)` — using `category` as a proxy for archetype. The external DB has a real `archetype` column. Change to `normaliseArchetype(service.archetype, service.name)` and simplify.

### 4. Fix `useServices` to filter by `is_active`
The external DB supports `is_active`. The `useServices` hook should filter `.eq('is_active', true)` and order by `sort_order`.

### 5. Verify Dashboard query works
Line 309 of `Dashboard.tsx` already selects `services(name, archetype)` — this should work against the external DB. No change needed here.

### 6. Verify MessageDrawer works
The `MessageDrawer` inserts `{ work_thread_id, sender_id, content }` into `messages`. The external DB's `messages` table has exactly these required columns. This should work as-is.

### 7. Verify work_thread creation (Phase 2 changes)
The `handleStartJob` function in `Dashboard.tsx` inserts into `work_threads` with `client_id`, `provider_id`, `job_request_id`. This matches the external DB schema. Should work.

## What's Already Working

- Job request creation (the `services(name, archetype)` join works on the external DB)
- Work thread creation on quote acceptance (Phase 2)
- Messaging via `messages` table with `content` column
